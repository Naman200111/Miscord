Critical Socket.io has no authentication
src/pages/api/socket/io.ts
▶

Any client — authenticated or not — can connect to the Socket.io server and emit chat:message events. The handler trusts the userId sent by the client directly in the message payload. A malicious user can impersonate any user ID or spam the DB with fabricated messages.

❌ Problem — userId comes from untrusted client payload
// io.ts — userId is whatever the client sends socket.on(`chat:message`, async (msgData: messageData) => { const { channelId, serverId, userId, msg, id } = msgData; // No check that userId == the logged in user await db.insert(messages).values({ id, channelId, serverId, userId, msg }); });
✅ Fix — verify Clerk session token on socket handshake
// Pass Clerk session token from client during connection const io = new Server(res.socket.server, { path: "/api/socket/io", }); io.use(async (socket, next) => { const token = socket.handshake.auth.token; try { const payload = await verifyToken(token, { secretKey: process.env.CLERK_SECRET_KEY }); socket.data.userId = payload.sub; // store verified userId on socket next(); } catch { next(new Error("Unauthorized")); } }); io.on("connection", (socket) => { socket.on("chat:message", async (msgData) => { const userId = socket.data.userId; // use verified server-side userId await db.insert(messages).values({ ...msgData, userId }); }); });

High Missing database indexes on foreign keys
src/db/schema.ts
▶

The schema defines foreign keys but no explicit indexes. Postgres does NOT auto-index FKs. Queries filtering on serverId, userId, channelId, and createdAt will do full table scans as data grows. For a chat app this will become noticeable very quickly.

✅ Add indexes to schema
import { index } from "drizzle-orm/pg-core"; export const messages = pgTable("messages", { ... }, (t) => ({ channelIdx: index("messages_channel_id_idx").on(t.channelId), serverIdx: index("messages_server_id_idx").on(t.serverId), createdAtIdx: index("messages_created_at_idx").on(t.createdAt), })); export const serverUsers = pgTable("server_users", { ... }, (t) => ({ userServerIdx: index("server_users_user_server_idx").on(t.userId, t.serverId), }));
High Using non-pooled Neon HTTP driver
src/db/drizzle.ts
▶

drizzle-orm/neon-http creates a new connection per query and is designed for edge runtimes. For a Next.js app with Socket.io running many concurrent queries, you want the WebSocket-based pooled driver to avoid connection limits and latency.

✅ Switch to neon-serverless with connection pooling
// npm install @neondatabase/serverless import { Pool } from "@neondatabase/serverless"; import { drizzle } from "drizzle-orm/neon-serverless"; const pool = new Pool({ connectionString: process.env.DATABASE_URL }); export const db = drizzle(pool);
